# Contrat API — v2.0 VFS Foundation (issues #120, #121)

> **Version** : 2.0.0-draft  
> **Date** : 2026-05-17  
> **Issues** : #120 (Drive WinFsp unifié), #121 (ReadAt + ChunkSize)  
> **Branche** : `feature/v2.0-vfs-foundation`

---

## 1. Résumé des changements

### 1.1 Drive WinFsp unifié (#120) — BREAKING

L'architecture v1.1.x crée un drive WinFsp distinct par backend (ex: E:, F:, G:).

L'architecture v2.0 monte un **drive unique GhD:** (lettre configurable dans `AppConfig.MountPoint`) exposant chaque backend comme **sous-dossier** :

```
GhD:\
├── MonNAS\          ← backend "MonNAS"
│   ├── docs\
│   └── photos\
├── WebDAV\          ← backend "WebDAV"
│   └── backup\
└── desktop.ini      ← virtuel (icône drive)
```

### 1.2 ReadAt + ChunkSize (#121) — BREAKING

Extension de l'interface `StorageBackend` avec deux nouvelles méthodes permettant
les range reads (hydratation progressive, cache de chunks).

## 2. Interface Go — StorageBackend (plugins/plugin.go)

### 2.1 Nouvelles méthodes

```go
// ReadAt reads up to length bytes from the remote file at the given byte offset.
// Returns the bytes read; len(data) may be less than length only on EOF.
// Used for range-reads enabling progressive hydration and chunk caching.
// Returns ErrFileNotFound (wrapped) when remote does not exist.
// Pre-condition: IsConnected() == true, else returns nil, ErrNotConnected.
ReadAt(ctx context.Context, remote string, offset, length int64) ([]byte, error)

// ChunkSize returns the natural I/O granularity for this backend in bytes.
// The chunk cache aligns range reads to this boundary.
// Returns 0 to request use of the global default chunk size.
// Immutable; may be called before Connect; must not perform I/O.
ChunkSize() int64
```

### 2.2 Valeurs ChunkSize attendues par plugin

| Plugin   | ChunkSize()              | Justification |
|----------|--------------------------|---------------|
| webdav   | 0 (défaut global)        | Pas de granularité native — l'offset HTTP Range est libre |
| moosefs  | 67_108_864 (64 MiB)      | Taille de chunk MooseFS native |
| local    | 0 (défaut global)        | Lecture POSIX libre |
| mock     | 0                        | No-op |

## 3. gRPC — storage.proto (plugins/proto/storage.proto)

### 3.1 Nouveaux messages

```protobuf
// ReadAt — range read (client unary → server unary avec payload bytes)
message ReadAtRequest {
  string remote_path = 1;
  int64  offset      = 2;
  int64  length      = 3;
}

message ReadAtResponse {
  bytes  data  = 1;  // bytes lus ; len(data) <= length
  string error = 2;  // non-vide si erreur terminale
}

// ChunkSize — retourne la granularité naturelle du backend
message ChunkSizeRequest {}

message ChunkSizeResponse {
  int64 chunk_size = 1;  // 0 = utiliser le défaut global
}
```

### 3.2 Nouvelles RPCs dans StorageService

```protobuf
rpc ReadAt    (ReadAtRequest)    returns (ReadAtResponse);
rpc ChunkSize (ChunkSizeRequest) returns (ChunkSizeResponse);
```

### 3.3 Numéros de champs réservés

- `ReadAtRequest` : champs 1–3 (ne pas réutiliser)
- `ReadAtResponse` : champs 1–2
- `ChunkSizeRequest` : aucun champ
- `ChunkSizeResponse` : champ 1

## 4. Architecture VFS — GhostFileSystem (internal/placeholder/)

### 4.1 router_windows.go — Routage multi-backend

```go
// route résout un chemin FUSE vers le backend propriétaire et le relPath interne.
// En v2.0, le premier segment du chemin identifie le backend par son Name.
// Exemples :
//   "/MonNAS"           → {backend: MonNAS, relPath: "/"}
//   "/MonNAS/docs/f.txt" → {backend: MonNAS, relPath: "/docs/f.txt"}
//   "/WebDAV/"          → {backend: WebDAV, relPath: "/"}
func (fs *GhostFileSystem) route(path string) *routeResult
```

**Algorithme :**
1. Extraire le premier segment après `/` : `backendName := segments[0]`
2. Rechercher dans `fs.backends` le backend dont `Config.Name == backendName` (insensible à la casse)
3. Construire `relPath = "/" + strings.Join(segments[1:], "/")`
4. Retourner `nil` si aucun backend ne correspond

### 4.2 filesystem_windows.go — Readdir root

`Readdir("/")` liste en v2.0 :
- Les fichiers virtuels `desktop.ini` + `ghostdrive.ico` (inchangés)
- Une entrée `S_IFDIR` par backend activé, avec le `Name` du backend comme nom de dossier

```go
// Getattr("/") → S_IFDIR | 0755 (inchangé)
// Readdir("/") → [desktop.ini, ghostdrive.ico, <backend1.Name>, <backend2.Name>, ...]
// Getattr("/<backendName>") → S_IFDIR | 0755 (virtuel)
// Getattr("/<backendName>/...") → déléguer au backend
```

### 4.3 Cas Getattr pour le dossier virtuel de backend

`Getattr("/<backendName>")` doit retourner `S_IFDIR | 0755` sans déléguer au backend
(le dossier est virtuel — il n'existe pas côté backend).

### 4.4 manager.go — DriveManager unifié

```go
// MountUnified monte le drive GhD: unique avec la liste de tous les backends activés.
// Si le drive est déjà monté avec les mêmes backends, c'est un no-op.
// Si le drive est déjà monté avec une config différente, il est remonté.
func (dm *DriveManager) MountUnified(mountPoint string, backends []MountedBackend) error

// UpdateBackends met à jour la liste des backends sur le drive unifié sans unmount.
// Utilisé quand un backend est ajouté/retiré dynamiquement.
func (dm *DriveManager) UpdateBackends(backends []MountedBackend) error

// UnmountUnified démonte le drive GhD:.
func (dm *DriveManager) UnmountUnified() error

// GetUnifiedStatus retourne le statut du drive unifié.
func (dm *DriveManager) GetUnifiedStatus() (DriveStatus, bool)
```

Les anciennes méthodes `Mount(backendID, mountPoint, mb)`, `Unmount(backendID)`,
`GetStatus(backendID)`, `GetAllStatuses()` sont **dépréciées** mais conservées
avec une implémentation de délégation pour éviter les breaking changes en app.go. 

### 4.5 GhostFileSystem — multi-backends

`GhostFileSystem.backends` passe de 1 backend à N backends.  
`fs.Statfs()` agrège les quotas de tous les backends (somme free, somme total).

### 4.6 watchLoop — par backend

Chaque backend possède sa propre goroutine `watchLoop`, inchangée en comportement.
Les événements émis sur le drive unifié incluent le backend ID dans `MetaUpdatedEvent`.

## 5. Configuration — AppConfig (internal/config/config.go)

`AppConfig.MountPoint` (existant, type `string`) est promu au rôle de **point de montage global** :

- Valeur par défaut : `"G:"` (première lettre disponible ≥ G)
- Ancienne valeur de chaque `BackendConfig.MountPoint` est **ignorée** pour la VFS
- `BackendConfig.MountPoint` devient déprécié (conservé pour compatibilité JSON)

## 6. Wails Bindings — Changements

### 6.1 GetDriveStatuses() — BREAKING

**Avant (v1.1.x)** : retourne `map[string]DriveStatus` — une entrée par `backendID`.

**Après (v2.0)** : retourne `map[string]DriveStatus` — une seule entrée, clé `"unified"`.

```go
// DriveStatus en v2.0 — source de vérité unique pour l'UI :
DriveStatus{
    Mounted:      true,
    MountPoint:   "G:",               // point de montage réel WinFsp
    BackendID:    "unified",
    BackendName:  "GhostDrive",       // == volname WinFsp == nom affiché dans le tray
    BackendPaths: map[string]string{
        "backendID1": `G:\MonNAS\`,   // chemin réel dans l'Explorateur Windows
        "backendID2": `G:\WebDAV\`,
    },
}
```

### 6.2 Règle de cohérence UI/drive — source de vérité unique

**`DriveStatus` est la seule source de vérité** pour l'affichage UI du drive.

| Donnée | Source | Interdit |
|--------|--------|---------|
| Nom du drive (tray) | `DriveStatus.BackendName` | Hardcoder "GhostDrive" dans le frontend |
| Point de montage | `DriveStatus.MountPoint` | Lire `AppConfig.MountPoint` côté frontend |
| Chemin d'un backend | `DriveStatus.BackendPaths[backendID]` | Reconstruire `mountPoint + "\\" + backendName + "\\"` |
| Liste des backends actifs | `DriveStatus.BackendPaths` (clés) | Lire `AppConfig.Backends` directement |
| État monté/démonté | `DriveStatus.Mounted` | État local frontend |

**Invariant garanti par le backend :**
- `DriveStatus.BackendName` == `volname` passé à WinFsp (`"GhostDrive"`)
- `DriveStatus.BackendPaths[id]` == chemin réel accessible dans l'Explorateur Windows
- `DriveStatus.MountPoint` == lettre de lecteur effective
- Les clés de `DriveStatus.BackendPaths` == exactement les backends accessibles dans `Readdir("/")` du drive WinFsp

**Côté backend (mount_windows.go)** :
```go
// Status() doit construire BackendPaths depuis les données réelles du drive :
paths[mb.ID] = mountPoint + `\` + mb.Name + `\`
// Exemple : "G:" + `\` + "MonNAS" + `\` = `G:\MonNAS\`
```

### 6.3 Règle de cohérence de contenu — UpdateBackends() comme point d'entrée unique

**`UpdateBackends()` est le seul point d'entrée** qui met à jour simultanément :
1. Le VFS WinFsp (sous-dossiers `Readdir("/")`)
2. L'état UI (liste des backends dans le tray via `DriveStatus.BackendPaths`)

**Invariants :**
- Backend activé → apparaît dans `GhD:\<Name>\` **ET** dans `DriveStatus.BackendPaths` **ET** dans le tray
- Backend désactivé → absent de `GhD:\` **ET** absent de `DriveStatus.BackendPaths` **ET** absent du tray
- Il n'existe aucun chemin dans `Readdir("/")` sans backend correspondant dans `DriveStatus.BackendPaths`, et vice versa

**Protocole `SetBackendEnabled` (app.go)** :
```
enable=true :
  1. manager.Add(bc)                  ← connecte le backend
  2. dm.UpdateBackends(newList)       ← met à jour VFS + DriveStatus atomiquement
  3. SI erreur → ne PAS persister Enabled=true dans AppConfig, retourner erreur au frontend
  4. SI succès → persister config, émettre drive:mounted

enable=false :
  1. dm.UpdateBackends(newList)       ← retire le backend du VFS en premier
  2. manager.Remove(bc.ID)           ← déconnecte ensuite
  3. persister config, émettre drive:unmounted
```

**Conséquence pour le frontend** :
- Ne jamais afficher un backend comme "activé" avant réception de `drive:mounted`
- Après `drive:mounted`, appeler `GetDriveStatuses()` pour obtenir `BackendPaths` mis à jour
- La liste affichée dans le tray doit être reconstruite depuis `DriveStatus.BackendPaths`, jamais depuis `AppConfig.Backends`

### 6.3 BackendConfig.MountPoint — DEPRECATED

Le champ `mountPoint` de `BackendConfig` n'est plus utilisé par la couche VFS.
Il est conservé dans le JSON de config pour compatibilité ascendante mais ignoré au montage.

## 7. Frontend (contracts/wails-events.md — mise à jour)

Événements inchangés : `drive:mounted`, `drive:unmounted`, `drive:error`, `meta:updated`.

Le payload de `drive:mounted` change :
```json
{
  "backendID": "unified",
  "backendName": "GhostDrive",
  "mountPoint": "G:"
}
```

**Le frontend doit récupérer `DriveStatus` complet via `GetDriveStatuses()` après `drive:mounted`**
pour obtenir `BackendPaths` — le payload de l'événement ne le contient pas.

## 9. Critères d'acceptation

### #120 — Drive WinFsp unifié
- [ ] Un seul drive WinFsp monté (ex: `G:`) même avec 2+ backends configurés
- [ ] `G:\MonNAS\` liste les fichiers du backend "MonNAS"
- [ ] `G:\WebDAV\` liste les fichiers du backend "WebDAV"
- [ ] Création de fichier dans `G:\MonNAS\docs\` → upload sur backend "MonNAS"
- [ ] Ajout/retrait d'un backend → `UpdateBackends()` sans remonter le drive
- [ ] `GetDriveStatuses()` retourne une seule entrée "unified"
- [ ] Tests unitaires routeur : routage correct pour 2 backends + ENOENT si inconnu
- [ ] `go test ./internal/placeholder/...` passe (≥70% couverture)
- [ ] **[Cohérence UI/drive]** `DriveStatus.BackendName` == `volname` WinFsp == nom affiché dans le tray (tous égaux à `"GhostDrive"`)
- [ ] **[Cohérence UI/drive]** `DriveStatus.BackendPaths["backendID"]` == chemin réel dans l'Explorateur Windows (ex: `"G:\\MonNAS\\"`)
- [ ] **[Cohérence UI/drive]** Le composant tray construit l'affichage exclusivement depuis `DriveStatus` — aucun chemin ni nom reconstructs côté frontend
- [ ] **[Cohérence contenu]** Activer un backend depuis l'UI → apparaît simultanément dans `GhD:\<Name>\` ET dans le tray — sans redémarrage
- [ ] **[Cohérence contenu]** Désactiver un backend depuis l'UI → disparaît simultanément de `GhD:\` ET du tray — sans redémarrage
- [ ] **[Cohérence contenu]** La liste affichée dans le tray est construite depuis `DriveStatus.BackendPaths`, jamais depuis `AppConfig.Backends` directement
- [ ] **[Cohérence contenu]** Si `UpdateBackends()` échoue côté VFS, le tray ne marque pas le backend comme activé (aucun état optimiste)
- [ ] **[Cohérence contenu]** Les clés de `DriveStatus.BackendPaths` == exactement les entrées de `Readdir("/")` (ni plus, ni moins)

### #121 — ReadAt + ChunkSize
- [ ] `ReadAt` sur WebDAV utilise l'en-tête HTTP Range
- [ ] `ReadAt` sur MooseFS utilise la lecture à offset natif
- [ ] `ReadAt` sur Local utilise `os.File.ReadAt`
- [ ] `ChunkSize()` retourne 0 pour webdav et local, 67_108_864 pour moosefs
- [ ] `GRPCBackend.ReadAt` + `GRPCBackend.ChunkSize` transmis via gRPC
- [ ] `GRPCServer` sert `ReadAt` et `ChunkSize`
- [ ] Tests unitaires pour `ReadAt` dans chaque plugin (WebDAV in-memory, MooseFS mock)
- [ ] `go test ./plugins/...` passe (≥70% couverture)
- [ ] Mock plugin et template compilent sans erreur
